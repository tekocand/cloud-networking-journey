### VPC Endpoints
- **Purpose:** Connect VPC to AWS services without traffic leaving AWS network
- **Without Endpoint:** Private subnet needs NAT Gateway to reach public AWS services
- **With Endpoint:** Traffic stays on AWS backbone, more secure and efficient

**Two Types:**

**1. Interface Endpoints (AWS PrivateLink)**
   - Creates ENI (Elastic Network Interface) in private subnet
   - Uses Route 53 DNS entry — AWS resolves to private ENI IP instead of public IP
   - Cost: ~$8.90/month per endpoint + $0.01/GB data transfer
   - Cost evaluation: If >4 services needed, compare with NAT Gateway ($37/month)
   - **HA Best Practice:** Create endpoint per AZ for redundancy
   - **Efficiency:** Configure instances to use endpoint in same AZ, others as backup

**2. Gateway Endpoints (FREE)**
   - Only for Amazon S3 and DynamoDB
   - Adds prefix list (all S3/DynamoDB public IPs) to VPC route table
   - Target is the gateway endpoint (not public IPs)
   - Traffic stays in AWS network — no data transfer cost
   - Most cost-effective for S3/DynamoDB access from private subnets

### VPC Flow Logs
- Captures IP traffic **metadata** (not actual payload data)
- Can be created at VPC, subnet, or network interface level (ELB, RDS, etc.)
- **Use cases:**
  - Troubleshoot connectivity issues (ACCEPT vs REJECT)
  - Monitor traffic patterns
  - Security analysis
  - Cost optimization (identify unwanted traffic)
- **Log format:** Shows src/dst IP, ports, protocol, packets, bytes, action (ACCEPT/REJECT)
- **Destinations:** CloudWatch Logs, S3, or Kinesis Data Firehose

### Network Manager
- Dashboard for monitoring and visualization of network topology
- Global network view across VPCs, Transit Gateways, and on-prem connections
- Helps visualize multi-VPC architectures and identify connectivity issues

## Major Troubleshooting Session

### Problem
Public instance (10.0.1.252) could not communicate with private instance (10.0.11.186)
- SSH timeout
- Ping "Destination Host Unreachable"
- Flow Logs showed ACCEPT inbound but REJECT outbound

### Root Cause
**Route table misconfiguration:**
- Had two separate private route tables
- One was missing `10.0.0.0/16 → local` route
- Private1 subnet was associated with the broken route table

### Solution
- Consolidated to **one private route table** for all private subnets
- Associated both private1 and private2 with the correct route table
- Restarted private instance to pick up new routes
- Connectivity restored ✅

### Key Learnings
1. **Keep route tables simple:** One for public, one for private
2. **Always verify route table associations** after creation
3. **Restart instances** after route table changes to renew DHCP lease
4. **Flow Logs are invaluable** for troubleshooting — showed ACCEPT vs REJECT clearly
5. **The `local` route is critical** — without it, instances can't communicate within VPC
